## 先说疼点：vibe coding 看多了，头会变大

很多 vibe coding 项目给我的感觉是：能跑，真的能跑；但你要问它为什么这么写、下一步怎么改、半年后谁来维护，那空气突然安静。

结构像临时搭起来的棚子：今天挡雨，明天加门，后天接电，月底突然说要装电梯。

不是不能住。

是住进去之后，每天都像在参加极限挑战。

我看到这些项目之后，脑子里冒出一个很朴素的念头：我不想再“读”这样的代码了，我想造一个能“写”出好项目的 Agent。

注意，不是写出一坨能跑的代码。

是写出一个可以继续维护、继续迭代、继续长大的项目。

  <p>代码生成器解决的是“这一段怎么写”。项目生成器要解决的是“这套东西以后怎么活”。</p>

## 为什么非要自己造一个

你可能会问：现成 Coding Agent 这么多，为什么还要自己造？

很现实。

一是不好用。二是贵。三是又贵又不一定好用。

Codex Plus 不够用，Codex Pro 太贵，Claude 还容易在关键时刻掉链子。每次看到价格，我都觉得自己不是在调用模型，是在给钱包做压力测试。

<BlogMeme
  src="/images/memes/agent-bill-wallet.jpg"

但真正让我想自己做的，不只是成本。

更核心的问题是：现在很多 Coding Agent 擅长“完成一次任务”，但不擅长“保护一个项目”。

第一次生成，很快。

第二次修改，还行。

第三次继续堆，开始有点不对劲。

第十次之后，项目像一碗被反复加水的粥：表面还是粥，底下已经不知道是什么了。

所以我想做的不是一个兴奋地一路往前冲的 Agent，而是一个懂得先停下来问清楚规格的 Agent。不是 vibe coding，而是 spec coding。

## 大模型是内核，不是许愿池

如果要造 Agent，第一件事不是堆工具，而是先承认一件事：

大模型很强，但它不是许愿池。

它的底层目标仍然是预测下一个 token。训练和后训练让它涌现出推理、规划、工具选择、代码生成这些能力，但它不会天然知道你的项目半年后会不会变成一锅粥。

这也是我这段时间最深的感受：

> 在缺少约束的时候，大模型倾向于把任务完成，而不是把任务做好。

“完成”很好判断。代码写完了，命令跑过了，页面打开了，掌声响起来。

“做好”就麻烦多了。它需要克制，需要回头看，需要考虑未来维护成本。说白了，它需要一点成年人的痛苦。

而模型没有这种痛苦。

所以不能指望一句“请写高质量代码”就解决问题。这就像跟熬夜的人说“早点睡”，道理很对，效果约等于没有。

## 1M 上下文也救不了一切

现在大家很喜欢讲长上下文。1M，上下文自由，听起来像把整个仓库塞进去就万事大吉。

但“能塞进去”和“能用得好”是两回事。

你把整个仓库怼给模型，它确实看到了。但看到了不代表理解了，理解了不代表修改时还记得，记得不代表不会在第七步突然开始自由发挥。

长上下文不是万能书包。它更像一个超大行李箱：什么都能装，但你在机场要找充电线的时候，还是会翻到怀疑人生。

真正有效的上下文，不是越多越好，而是在正确的时间给它正确的信息。

这也是为什么我越来越倾向于把 Agent 的工作流拆开：每一步只给它当前必须知道的东西，让它少点自由发挥，多点按图施工。

## ReAct 能跑，但不保证质量

现在很多 Coding Agent，大体都是 ReAct 思路：模型思考，调用工具，观察结果，再继续行动。

这套机制可以工作，但它不天然保证工程质量。

因为 Agent 在任务中最容易追逐的是“眼前闭环”：这个报错修了，这个页面出来了，这个测试过了。

但项目真正的风险往往藏在更后面：职责边界模糊、重复实现、依赖关系打结、抽象过早、错误处理缺失。

这些问题不像编译错误一样当场拍桌子。

它们更阴险：今天不疼，明天不疼，下个月突然让你在凌晨三点打开电脑。

## 第一种病：结构失控

结构失控是 Coding Agent 最常见的病。

一开始只是为了快，随手加一点逻辑。后来需求变了，再补一点判断。再后来功能多了，又塞一个参数。最后一个文件开始承担路由、状态、校验、请求、展示、缓存、人生哲学。

这时候你再问它职责是什么，它沉默。

我觉得代码和写书很像。没有结构的代码，就像一本章节乱飞的书：第一章剧透结局，第二章重复序言，第三章突然开始教你煎蛋。

读者很累。

未来的开发者更累。

未来的 Agent 也累。它不是不想帮你，是它看完也想请假。

  <p>好的结构不是为了显得专业，而是为了让下一次修改不需要先进行考古。</p>
</ArticleInsight>

## 第二种病：重复造轮子

另一种病是重复造轮子。

Agent 如果不知道项目里已经有什么，就很容易凭空再写一套。明明已有模块可以复用，明明成熟库已经解决，它偏要现场表演一个“我也可以”。

结果通常是：它确实可以，但你不一定敢用。

这就像本来“风和日丽”四个字能说清，它非要写成：

> 今日处于日照条件良好、云量较低、气温适中、空气流动柔和、体感舒适且适宜户外活动的综合气象状态。

看起来很努力。

实际上是在给中文增加技术债。

所以我的项目生成器必须把“先查已有实现”变成流程，而不是建议。

## 提示词不够，得上工程约束

提示词当然重要，但提示词不是保险柜。

你可以在提示词里写一百遍“不要重复造轮子”“保持单一职责”“写完要测试”，但模型执行到中后段时，还是可能开始凭感觉补丁式前进。

所以我更想做的是工程约束。

不是“建议你这样做”，而是“流程要求你必须这样做”。

比如：

- 修改代码前，必须检索项目已有模块；
- 新增抽象前，必须说明它减少了什么复杂度；
- 改完代码后，必须跑验证；
- 重要变更后，必须有独立审查；
- 遇到阻碍时，必须先诊断首选路线，而不是直接换条路跑路。

这些步骤听起来慢，但其实是在省未来的时间。

不然今天省下来的十分钟，迟早会变成未来某个下午的三个小时，外加一杯冰美式和一句“我当时为什么要这么写”。

## 好用第一，省钱第二

做 Agent 的时候，成本当然要考虑。prompt cache 要考虑，模型价格要考虑，工具调用次数也要考虑。

但我的排序很明确：

**好用第一，省钱第二。**

一个不好用但很省钱的 Agent，本质上是在用低价购买卡住。

我想要的是能稳定推进项目的人机协作系统，不是一个每次都让我重新解释人生的聊天窗口。

当然，便宜也很重要。毕竟 DeepSeek V4 如果足够好，那谁不想拥有一个“价格友好型队友”呢？

## 项目也应该为 Agent 设计

过去项目文档主要写给人看：README、架构图、接口文档、开发规范。

但 Agent 进入开发流程之后，项目也应该有一部分内容写给 Agent 看。

比如：

- 模块职责说明；
- 已有能力索引；
- 常见坑位记录；
- 决策记录；
- 验收标准；
- 不允许碰的边界。

这些不是形式主义。

对人来说，它们是文档；对 Agent 来说，它们是导航。

没有导航的 Agent，就像第一次去大型商场停车场找车的人：它也很努力，但方向感全靠命运。

## 单 Agent 不够稳，多 Agent 才像工程

如果只是一次性写个小工具，单 Agent 可能够用。

但如果目标是长期维护一个项目，我越来越觉得单 Agent 不够稳。

更合理的方式，是多 Agent 协作：一个负责实现，一个负责审查，一个负责测试，一个负责沉淀经验。每个 Agent 只盯住自己的职责，不要让一个上下文同时扮演开发、评委、测试和项目经理。

这听起来像在给 Agent 开会。

但软件工程不就是这样吗？人类团队也是靠分工减少盲区，只不过 Agent 团队更便宜，也不会在日报里写“今日主要熟悉项目”。

我设想里的项目生成器，应该先把项目拆成模块，理清依赖关系，再从叶子节点开始并行开发。

模块之间通过清晰接口连接，依赖尽量单向流动。允许必要的受控冗余，但不允许无知型重复。

换句话说：不是为了消灭所有重复，而是为了避免项目依赖打结。

## 我真正想造的是什么

到这里，答案就比较清楚了。

我想造的不是一个“帮我写代码”的 Agent。

我想造的是一个会管住自己的项目生成器。

它应该知道什么时候该写，什么时候该先查；什么时候该复用，什么时候该抽象；什么时候该继续修首选路线，什么时候才该承认这条路暂时走不通。

它不应该只追求“任务完成”，而应该追求：

> 在不破坏长期质量的前提下，把任务完成。

这句话听起来没那么热血，但很实用。

毕竟真正能陪项目走到最后的，从来不是一时兴起的炫技，而是稳定、克制、可复用的工程习惯。

## 写在最后

如果上篇是“别急着造 Agent，先学会用好 Agent”，那这篇就是我终于忍不住往前走了一步：

我不是想造一个更会 vibe coding 的 Agent。

我想造一个能把 vibe coding 按在椅子上、让它先写 spec 的 Agent。

下次再聊具体怎么做。

希望那时候，我的钱包和上下文窗口都还活着。
